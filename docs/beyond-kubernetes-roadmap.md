# Beyond Kubernetes 方向

本文档记录 Boss 的长期产品与架构方向。当前系统架构见[系统架构设计](system-architecture.md)，控制器体系见[控制器架构设计](controller-architecture.md)，实施路线见[实施路线](implementation-roadmap.md)，设计原则见[设计哲学](design-philosophy.md)。

Boss 应与通用容器编排走不同的竞争路线。目标是把异构沙箱执行作为一等能力：进程、容器、微虚拟机、完整虚拟机、WASM 和远程执行环境共享同一套控制面，而不是被硬塞进容器模型。

## 产品论点

系统应优化那些在同一集群中需要不同隔离、启动速度、成本和可移植性取舍的工作负载。用户应当一次提交运行意图，然后让平台根据策略、能力和运行时健康状态选择最合适的执行底层。

实际目标不是与成熟编排系统实现功能对等。目标是一个更简洁、更锐利的沙箱原生调度平台：

- 多种沙箱类型是一等能力，不是插件的边缘场景。
- 隔离策略是可调度、可观察、可执行的。
- 运行时能力由节点上报并由调度器使用。
- 启动延迟和制品本地化纳入放置决策。
- 失败原因在所有 Provider 之间标准化。
- API 保持足够小，新增 Provider 不需要修改每个子系统。

## 差异化方向

### 1. 沙箱原生的 API

当前容器优先的 API 让非容器工作负载感觉像事后补丁。Boss 应围绕执行意图建模工作负载：

- `sandboxClass`：请求的执行类别，例如 `process`、`container`、`microvm`、`vm`、`wasm`、`remote`。
- `artifact`：可运行单元，例如可执行文件、OCI 镜像、WASM 模块、rootfs 镜像、磁盘镜像或远端模板。
- `isolation`：所需边界，例如宿主共享、命名空间隔离、运行时沙箱、内核隔离或硬件虚拟化。
- `network`、`storage`、`security`：明确的声明式需求，而不是运行时专用标记。

这为用户、调度器、节点代理和 Provider 之间建立了稳定契约。

### 2. 基于能力的调度

调度应由节点真实能力驱动，而不是假设每个节点能力相同。

需要的调度器输入：

- 每个节点上的健康 Provider。
- 支持的沙箱类别和制品类型。
- 支持的隔离级别、网络模式、存储模式和设备。
- Provider 特定的容量，例如微虚拟机槽位或 WASM 并发数。
- 制品缓存状态和启动延迟提示。

需要的调度器行为：

- 过滤无法满足请求沙箱契约的节点。
- 根据资源匹配度、冷启动成本、制品本地化、Provider 健康和隔离匹配度对节点评分。
- 在绑定前预留稀缺的 Provider 容量。
- 将选中的 Provider 记录在 Pod 注解中，使节点代理能做确定性选择。

### 3. 强语义的运行时契约

Provider 集成应兼具乏味和严格。每个 Provider 应实现一个小的契约，附带清晰的幂等性和错误语义。

需要的 Provider 能力：

- `capabilities()`：描述支持的类别、制品、隔离、网络、存储、设备和容量。
- `prepare()`：在 create/start 之前获取和校验制品。
- `create()`、`start()`、`stop()`、`remove()`：具有幂等行为的生命周期操作。
- `status()`：返回标准化状态和 Provider 详细信息。
- `logs()` 和 `metrics()`：暴露统一的观测表面。

关键改进不是添加更多钩子。关键改进是让每个钩子在所有沙箱类型之间含义一致。

### 4. 隔离作为一等调度维度

隔离应该是调度器能推理的事情，而不是选中运行时后的副作用。

推荐的隔离级别：

- `shared-host`：宿主进程，低隔离。
- `namespaced`：命名空间和 cgroups。
- `runtime-sandboxed`：WASM 或用户态沙箱。
- `kernel-isolated`：独立客户内核或等价边界。
- `hardware-virtualized`：硬件虚拟化边界。

规则：

- 工作负载可以请求最低隔离级别。
- Provider 声明它能实际提供的隔离级别。
- 策略允许时，调度器可以将工作负载放置在更强的隔离级别上。
- 如果选中的 Provider 不能满足已解析的隔离策略，节点代理必须拒绝该工作负载。

### 5. 通过制品感知加速冷启动

平台应将制品准备视为可调度的成本。

设计要点：

- 引入 `ArtifactResolver`，将 URI 转换为本地已准备的制品。
- 跟踪每个节点和 Provider 的缓存状态。
- 将缓存命中信息反馈到调度器评分中。
- 支持已知工作负载的预拉取。
- 标准化制品错误，例如认证失败、摘要不匹配、不支持的格式、拉取超时。

这对 WASM 模块、微虚拟机 rootfs 镜像和大尺寸 VM 磁盘镜像尤为重要。

### 6. 更好的失败解释

用户不应需要 Provider 专属知识来理解常见故障。

标准化原因：

- `UnsupportedClass` — 不支持的沙箱类别
- `UnsupportedArtifact` — 不支持的制品类型
- `IsolationMismatch` — 隔离级别不匹配
- `ProviderUnavailable` — Provider 不可用
- `ArtifactFetchFailed` — 制品获取失败
- `InvalidSpec` — 请求规格无效
- `InsufficientResources` — 资源不足
- `PermissionDenied` — 权限拒绝
- `CreateFailed` — 创建失败
- `StartFailed` — 启动失败
- `StatusUnknown` — 状态未知

Provider 特定的详细信息应保留在结构化元数据中，但顶层 reason 必须足够稳定以支持自动化。

### 7. 最小核心，可扩展边缘

控制面应避免因 Provider 细节而膨胀。Provider 专属逻辑应放在 Provider 契约和声明式能力之后。

核心负责：

- 对象存储和 watch。
- 工作负载意图 API。
- 调度流水线。
- 节点能力模型。
- 生命周期状态机。
- 标准化的事件、日志、指标和错误。

Provider 负责：

- 制品格式细节。
- 运行时特定的 create/start/stop 行为。
- 运行时特定的日志和指标采集。
- 运行时依赖检查。
- 本地垃圾回收细节。

## 架构原则

1. 让沙箱类型成为调度输入，而不是运行时的事后想法。
2. 优先使用显式契约，而非隐式运行时行为。
3. 保持 Provider API 稳定且窄小。
4. 将启动延迟和制品本地化作为放置信号。
5. 从观测到的节点状态报告能力，而不是静态假设。
6. 为本地开发保留一条简单的默认路径。
7. 每个新资源都设计为可 watch 和可 reconcile。

## 近期工程计划

### 重定位与建模

- 更新公共文档，将系统描述为沙箱原生编排。
- 为 `sandboxClass`、沙箱需求和运行时能力增加 API 结构体。
- 保留 `runtimeClass` 作为兼容字段。
- 为进程、WASM、容器和微虚拟机工作负载添加示例。

验收标准：

- 用户无需 Provider 专属字段即可表达沙箱意图。
- 节点能上报至少一组 Provider 能力。

### 能力感知的节点代理

- 扩展 `RuntimeManager`，支持按名称和类别索引 Provider。
- 在运行时 crate 中增加 `RuntimeCapabilities`。
- 让节点代理在 `NodeStatus` 中发布 Provider 能力。
- 从调度器注解或本地默认值确定性解析 Provider 选择。

验收标准：

- 一个节点可以公布多个 Provider。
- 节点代理能用标准化原因拒绝不兼容的工作负载。

### 调度器过滤与 Provider 绑定

- 实现按沙箱类别、制品类型、隔离级别、网络模式和资源的调度过滤。
- 增加制品本地化和 Provider 健康的评分。
- 在绑定阶段写入选中的 Provider 注解。

验收标准：

- 工作负载只绑定到能实际运行它的节点。
- 选中的 Provider 能从 `bossctl get pods -o wide` 中可见。

### 制品解析器

- 为可执行文件、镜像、WASM 模块、rootfs、磁盘镜像和远端模板制品引入解析器接口。
- 增加本地缓存元数据。
- 标准化制品获取和校验错误。

验收标准：

- Provider 接收到本地已准备的制品。
- 调度器可以对缓存命中评分。

### 真实 Provider

- 完成进程 Provider 的语义。
- 增加最小 WASM Provider。
- 增加最小容器 Provider。
- 在可选 feature 后增加最小微虚拟机 Provider。

验收标准：

- 同一 API 能端到端启动至少三种沙箱类别。
- 日志、状态、退出码和失败原因保持一致。

## 设计准入门槛

提议的功能在进入核心前应通过以下检查：

- 是否适用于多种沙箱类型？
- 能否表达为能力、策略、状态或制品元数据？
- 调度器能否在不知晓 Provider 内部实现的情况下推理它？
- 节点代理能否在尝试启动工作负载之前校验它？
- 如果失败，用户能否看到稳定的失败原因？

如果答案大多是否定的，该功能应放在 Provider 内部，而不是核心 API。
